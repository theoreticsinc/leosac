10 Introduction {#mod_doorman_intro}
11 =================================
12 
13 The Doorman module is very important to Leosac's usability.
14 It is the module responsible for reacting to messages sent by authentication / authorization modules.
15 
16 If an access is granted (or denied) one (or more) actions can be taken by this module:
17 + Open the door.
18 + Raise an alarm.
19 + ...
20 
21 The main goal of this module is to trigger action on others components in the system. This is done
22 by sending message to them.
23

50 @note The `<cmd>` tag is quite simple. It looks like this:
51 ~~~~~~~~~~~~~~~~~~~~~~~.xml
52 <cmd>
53  <f1>GREEN_LED</f1>
54  <f2>ON</f2>
55  <f3>1500</f3>
56 </cmd>
57 ~~~~~~~~~~~~~~~~~~~~~~~
58 For each frame you want in the command message, use `<fX>` where `X` is growing for each frame you want.
59 
60 @hr
61 
62 @note `instances.instance.actions.action.target` is the name of an object that has been defined in the system.
63 It could be the name of a GPIO you defined before ("my_super_gpio")
64 or some reader object ("my_wiegand1"), etc.
65 
66 @hr
67 
68 @note Declaring `doors` is optional, and is only ever useful if you make use of
69 the "always open" or "always close" feature.

71 Example 0 {#mod_doorman_example_0}
72 ----------------------------------
73 
74 This is a very simple example that requires a component named "gpio_door" to open
75 for 2.5 seconds when an successful authentication happens. If the authentication fails,
76 it would ask the "led_failure" to turns on to 2.5s.
77 
78 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
79 <module>
80  <!-- Doorman module: react to authentication broadcast -->
81  <name>DOORMAN</name>
82  <file>libdoorman.so</file>
83  <level>50</level>
84  <module_config>
85  <instances>
86  <instance>
87  <name>DOORMAN_JARVIS_1</name>
88  <auth_contexts>
89  <auth_context>
90  <name>AUTH_CONTEXT_1</name>
91  </auth_context>
92  </auth_contexts>
93  <actions>
94  <action>
95  <on>DENIED</on>
96  <target>led_failure</target>
97  <cmd>
98  <f1>ON</f1>
99  <f2>2500</f2>
100  </cmd>
101  </action>
102  <action>
103  <on>GRANTED</on>
104  <target>gpio_door</target>
105  <cmd>
106  <f1>ON</f1>
107  <f2>2500</f2>
108  </cmd>
109  </action>
110  </actions>
111  </instance>
112  </instances>
113  </module_config>
114 </module>

118 Example 1 {#mod_doorman_example_1}
119 ----------------------------------
120 
121 The following example file defines the following:
122  + The door is always open on wednesday, between 13:13 and 13:15.
123  + The door is always closed on wednesday, between 13:15 and 15:16.
124  + On successful authentication (except when always open/close) we
125  send the `GREEN_LED` (with `ON` and `1500` as parameters) command to the
126  `MY_WIEGAND_1` device. This will light the green led of the reader for 1.5second.
127  + On failure (again, except when always open/close) we send the `ON` command (with `500` as
128  parameter) to the `wiegand_buzzer` GPIO, causing the device to beep for 0.5 second.
129
