14 Introduction {#auth_intro}
15 =============================
16 
17 In an access control system, the authentication & authorization subsystem is complex.
18 This page aims to explain how it works in Leosac.
19 
20 The task of reading credentials, validating them and taking action (opening doors) is split
21 over multiple module. This gives us more flexibility and helps keeping the code clear and relatively simple.
22 
23 Below is a diagram that illustrate how thing works:
24 ![Auth overview diagram](../resources/overview_auth.png)

28 Two kinds of modules
29 ====================
30 
31 We break down the authentication & authorization code in multiple parts.
32 There are two kinds of modules:
33  1. Authentication Source module (eg [Wiegand](@ref mod_wiegand_main))
34  2. Authentication Backend module (eg [Files](@ref mod_auth_file_main))
35 
36 Authentication Source
37 ---------------------
38 
39 An **authentication source** module provides user-data to the system, for example
40 a card id or a pin code. Theses modules are first in the chain of event that lead
41 to an access control.
42 
43 An authentication **shall** have at least one property: a name.
44 This name will be used throughout the application to determine what device (or authentication
45 source) triggered the events.

50 Authentication Backend
51 ----------------------
52 
53 An **authentication backend** module gather event from **authentication source**
54 and check their own backend to build a profile (raw files, database, remote webservices, ...)
55 
56 The profile is built by checking group-membership, credentials mapping, etc.
57 The newly built profile is then queried to determine whether or not the user should be granted access.
58 
59 In order for this to work properly, we need a decent message passing specifications.
60 
61 Authentication data type {#auth_data_type}
62 ------------------------------------------
63 
64 Data type information helps authentication backends understand messages from authentication sources.
65 It is an enumeration value, see [AuthSourceType](@ref Leosac::Auth::SourceType).
66 
67 Currently we are lucky, we only define a few.
68 
69 1. **SIMPLE_WIEGAND**: This data type requires 2 frames:
70  1. An hexadecimal *string* that contain the card id that was read, for example: "40:a0:83:80"
71  2. An *integer* that holds the number of bits in the card id.
72 2. **WIEGAND_PIN**: Configuration mode used when we use the reader only for PIN code.
73  1. A *string* that contains the PIN code. Its easier to use string, because we don't know the length
74  of the PIN code, and it may vary.
75 3. **WIEGAND_CARD_PIN**: This credential mode holds both a wiegand card number along with a PIN code.
76  1. An hexadecimal *string* that contain the card id that was read, for example: "40:a0:83:80"
77  2. An *integer* that holds the number of bits in the card id.
78  3. A *string* that contains the PIN code.
79 4. **SIMPLE_CSN**: A Card Serial Number in hexadecimal string format, separated by ":"
80  1. This Auth::SourceType requires a single frame. Example: "aa:bb:42:cd"

83 Message Passing Specifications {#auth_specc}
84 ============================================
85 
86 Understand the [application message bus](@ref MessageBus) first.
87 
88 First, a few assumptions. Authentication data from auth source may vary widely.
89 Consider a simple pin code versus a wiegand card id. The wiegand card must holds 2 data: the value, and the
90 number of meaningful bits.
91 A wiegand reader can also handle PIN code + card id, augmenting complexity of the auth data.
92 
93 Auth backend must be able to distinguish between those to handles them properly.
94 
95 ### Authentication Source
96 
97 Workflow for an **authentication source** module:
98  1. Having enough meaningful data that represents an authentication source (card id, pin code, fingerprint, ...)
99  the module shall publish a message on the message bus the let others know.
100  2. The message shall be multiparts.
101  1. Its first frame must be the auth source's name prefixed by `S_`.
102  2. The second frame shall be the type of data. See [this](@ref auth_data_type).
103  3. Optional (but likely presents) frames that contains different data based on the [auth data type](@ref auth_data_type)
104 
105 
106 As an example, consider a message from the [wiegand module](@ref mod_wiegand_main), named `MY_WIEGAND_1`:
107 
108 Frame | Content | Type
109 ---------|-----------------------------------------------|-------------------------------------------------------------
110 1 | "MY_WIEGAND_1" | `string`
111 2 | `Leosac::Auth::SourceType::SIMPLE_WIEGAND` | [AuthSourceType](@ref Leosac::Auth::SourceType) (`uint8_t`)
112 3 | "12:af:cd:21" | `string`
113 4 | 26 | `int`
114 
115 
116 A second example for the `WIEGAND_PIN` auth source mode:
117 
118 Frame | Content | Type
119 ---------|-----------------------------------------------|-------------------------------------------------------------
120 1 | "MY_WIEGAND_1" | `string`
121 2 | `Leosac::Auth::SourceType::WIEGAND_PIN` | [AuthSourceType](@ref Leosac::Auth::SourceType) (`uint8_t`)
122 3 | "1234591" | `string`

125 ### Authentication Backend
126 
127 Workflow for an **authentication backend** module:
128  1. Receive a message from an **auth source**.
129  2. Process the message, and decide whether access shall be granted or not.
130  3. Send a multiparts response message.
131  1. The first frame MUST be `S_{AUTH_INSTANCE_NAME}`.
132  2. The second frame shall be an [AccessStatus](@ref Leosac::Auth::AccessStatus).
133 
134 Frame | Content | Type
135 ---------|-----------------------------------------------|-------------------------------------------------------------
136 1 | "S_MY_AUTH_SOURCE_1" | `string`
137 2 | `Leosac::Auth::AccessStatus::Granted` | [AccessStatus](@ref Leosac::Auth::AccessStatus) (`uint8_t`)
