13 Goals {#install_guide_goals}
14 ============================
15 
16 The goals of this tutorial is to end-up with a simple and working setup.
17 
18 You'll have the ability to control a **door**. A **card reader** will read your
19 input card, and Leosac (running on a **Raspberry Pi**) will take care of orchestrating
20 the whole thing.
21 
22 
23 Requirements {#install_guide_req}
24 =================================
25 
26 Here is a list of requirements, both hardware and software.
27 The hardware requirements hard requirements for this tutorial to work.
28 
29 Hardware {#install_guide_req_hard}
30 ----------------------------------
31 
32 1. A Raspberry Pi (and a SD card).
33 2. A Piface Digital board (see [here](http://www.piface.org.uk/products/piface_digital/)).
34 3. A Wiegand card reader.
35 4. A Door.
36 
37 Software {#install_guide_req_soft}
38 ----------------------------------
39 
40 1. A SD card with a dumped Raspbian image.
41 2. GCC 4.8+ (any C++11-compatible compiler should do)
42 3. Git (to clone the repo)
43 4. CMake 2.8.12 (and above)
44 5. Boost 1.41 (and above)
45 6. TCLAP
46 7. GoogleTest is required if you plan to build the test suite.
47 
48 
49 **Note:** Leosac will work with something else than Raspbian, but some steps may change.
50 If you know what you are doing and are fluent with Linux you shouldn't have any problem following this guide even if you don't use Raspbian.

52 Getting Started {#install_guide_getting_started}
53 ================================================
54 
55 There is 2 thing to make sure before installing Leosac:
56 1. You have an updated system. Run `apt-get update && apt-get upgrade` to make sure of this.
57 2. The SPI kernel module is installed. The PifaceDigital card needs this module to work.\n
58  On Raspbian you can turn it on using `raspi-config` --> `Advanced Options` --> `SPI`.\n
59  Otherwise you can try to manually load the module: `modprobe spi_bcm2708`.

74 Debian Package {#install_guide_installing_deb}
75 ----------------------------------------------
76 
77 Grab the latest armhf package from the [Github Release tab](https://github.com/leosac/leosac/releases).
78 Install the package and install missing dependencies:
79 
80 ~~~~~~~~~~.sh
81 apt-get update;
82 dpkg -i leosac_armhf.deb;
83 apt-get install -f;
84 ~~~~~~~~~~
85 
86 Before going to the next step, make sure you can launch the `leosac` binary (ie, that it is
87 available in your PATH).

89 Basic Configuration {#install_guide_basic_cfg}
90 ==============================================
91 
92 Okay, so we now have Leosac installed. It needs to be configured before being useful.
93 Below is a base configuration file:
94 
95 ~~~~~~~~~~~~~~~~~~~~~.xml
96 <?xml version="1.0"?>
97 <kernel>
98  <instance_name>my_leosac</instance_name>
99  <plugin_directories>
100  <plugindir>/usr/lib/leosac</plugindir>
101  <plugindir>/usr/lib/leosac/auth</plugindir>
102  </plugin_directories>
103  <network>
104  <enabled>false</enabled>
105  </network>
106  <modules>
107  <module>
108  <name>MODULE_NAME</name>
109  <file>SHARED_LIBRARY_FILE</file>
110  <level>41</level>
111  <module_config>
112  <!-- Specific module configuration -->
113  </module_config>
114  </module>
115  </modules>
116 </kernel>
117 ~~~~~~~~~~~~~~~~~~~~~
118 
119 Explanations:
120 + `plugin_dir` is a folder use to try and find Leosac's dynamic libraries (aka modules). This path will NOT be
121  search recursively.\n
122 + `network` configure the network. Unless you need to, we recommend you leave it disabled and configure
123  networking through your system utility (`/etc/network/interfaces` or whatever you use).
124 + For modules:
125  1. The `name` is at your discretion, but it make to use something close to the shared library file.
126  2. `file` is the name of the shared library (`.so`) file.
127  3. `level` is the order of module loading. The closer to 0 it is, the sooner the module will be load.
128  4. `module_config` is for per-module specific configuration information.
129 
130 **MAKE SURE YOU DONT USE THE SAME `LEVEL` TWICE**

134 Adding modules {#install_guide_add_mod}
135 =======================================
136 
137 We need a few modules to make thing works:
138 + Since we are using the PifaceDigital card, we need Leosac's [piface digital](@ref mod_piface_main) module.
139 + We also need the [wiegand module](@ref mod_wiegand_main) to interact with the card reader.
140 + The [file auth](@ref mod_auth_file_main) module will be used to grant (or deny) access after receiving data
141  rom the card reader. This module will also store our card and time slice configuration.
142 + Last but not least, we need the [doorman](@ref mod_doorman_main) module to react to an authentication attempt.
143 
144 We will do this step by step, as modules configuration will depends on our hardware.
145 
146 Configuring the Piface Digital module {#install_guide_add_mod_piface}
147 ---------------------------------------------------------------------
148 
149 You can find the documentation of the module [here](@ref mod_piface_main). It focus on how to configure
150 the module for any use case. We will instead explain why we need a given configuration.
151 
152 We are using a Wiegand card reader. Therefore we need 2 INPUTs GPIO.
153 We also have 1 OUTPUT gpio for the door.
154 
155 We add this to our config file:
156 
157 ~~~~~~~~~~~~~~~~~~~.xml
158 <module>
159  <name>PIFACEDIGITAL_GPIO</name>
160  <file>libpifacedigital.so</file>
161  <level>2</level>
162  <module_config>
163  <gpios>
164  <gpio>
165  <name>my_door_gpio</name>
166  <no>3</no>
167  <direction>out</direction>
168  <value>false</value>
169  </gpio>
170  <gpio>
171  <name>wiegand_data_high</name>
172  <no>4</no>
173  <direction>in</direction>
174  </gpio>
175  <gpio>
176  <name>wiegand_data_low</name>
177  <no>2</no>
178  <direction>in</direction>
179  </gpio>
180  </gpios>
181  </module_config>
182 </module>
183 ~~~~~~~~~~~~~~~~~~~
184 
185 As you can see, we are defining GPIO using the gpio port `<no>` we wired our reader and door.
186 !IMAGE HERE!
187 
188 Note that we use `<level>2</level>` to make sure this module is one of the first to start.
189 We are assigning meaningful name to our GPIO. Keep them in mind because we will reuse them later.
190 
191 Configure the Wiegand module {#install_guide_add_mod_wiegand}
192 -------------------------------------------------------------
193 
194 Now that our configured GPIO are available to the rest of the application, we can configure the
195 module that will understand our Wiegand Card Reader.
196 
197 We have 1 reader to configure, and we will use both INPUT pin: `wiegand_data_low` and `wiegand_data_high`.
198 
199 ~~~~~~~~~~~~~~~~~~~.xml
200 <module>
201  <name>WIEGAND</name>
202  <file>libwiegand.so</file>
203  <level>5</level>
204  <module_config>
205  <readers>
206  <reader>
207  <name>MY_WIEGAND_1</name>
208  <high>wiegand_data_high</high>
209  <low>wiegand_data_low</low>
210  <green_led></green_led>
211  <buzzer></buzzer>
212  </reader>
213  </readers>
214  </module_config>
215 </module>
216 ~~~~~~~~~~~~~~~~~~~
217 
218 As you can see, we are re-using `wiegand_data_low` and `wiegand_data_high`. We assign them to our
219 reader (`MY_WIEGAND_1`).
220 **Note**: Most wiegand card readers also have a green led and a buzzer than you can activate. We are not using
221  them in this tutorial, we are leaving them out in the configuration.
222 
223 Configure the auth-file module {#install_guide_add_mod_auth_file}
224 -----------------------------------------------------------------
225 
226 In every access control system we need a component that is able to determine
227 if an access should be granted or denied. This is what the [auth file](@ref mod_auth_file_main) module
228 is doing.
229 
230 For this module we need a new configuration file, dedicated to this module only. This file will holds
231 cards informations, group and user mapping and time slice informations.
232 
233 Lets say this file is named "auth.xml".
234 
235 ~~~~~~~~~~~~~~~~~~~.xml
236 <root>
237  <users>
238  <user>
239  <name>MY_USER</name>
240  </user>
241  </users>
242  <credentials>
243  <map>
244  <user>MY_USER</user>
245  <WiegandCard>
246  <card_id>80:83:a0:40</card_id>
247  <bits>32</bits>
248  </WiegandCard>
249  </map>
250  <map>
251  <user>MY_USER</user>
252  <PINCode>
253  <pin>1234</pin>
254  </PINCode>
255  </map>
256  </credentials>
257  <schedules>
258  <schedule>
259  <name>all</name>
260  <!-- Has full access on monday and sunday -->
261  <monday>
262  <start>00:00</start>
263  <end>23:59</end>
264  </monday>
265  <tuesday>
266  <start>00:00</start>
267  <end>23:59</end>
268  </tuesday>
269  <wednesday>
270  <start>00:00</start>
271  <end>23:59</end>
272  </wednesday>
273 
274  <sunday>
275  <start>00:00</start>
276  <end>23:59</end>
277  </sunday>
278  </schedule>
279  </schedules>
280  <schedules_mapping>
281  <map>
282  <schedule>all</schedule>
283  <user>MY_USER</user>
284  </map>
285  </schedules_mapping>
286 </root>
287 ~~~~~~~~~~~~~~~~~~~
288 
289 For more information about the authentication and authorization config file, you can
290 read [this](@ref mod_auth_file_main).
291 
292 To enable this module, we also need to add something in the main configuration file. This something
293 looks like this:
294 
295 ~~~~~~~~~~~~~~~~~~~.xml
296 <module>
297  <name>AUTH_FILE</name>
298  <file>libauth-file.so</file>
299  <level>41</level>
300  <module_config>
301  <instances>
302  <instance>
303  <name>AUTH_CONTEXT_1</name>
304  <auth_source>MY_WIEGAND_1</auth_source>
305  <config_file>auth.xml</config_file>
306  <target>doorA</target>
307  </instance>
308  </instances>
309  </module_config>
310 </module>
311 ~~~~~~~~~~~~~~~~~~~
312 
313 The `<target>` match `<door>` in the `auth.xml` file. `<auth_source>` is the name of the reader
314 that will generate access attempt.
315 
316 When an attempt is detect, this module will analyze it and either grant the access, or deny it. This response
317 is then read by the doorman module.
318 
319 Configuring the doorman module {#install_guide_add_mod_doorman}
320 ---------------------------------------------------------------
321 
322 At last, we need to react to those "grant access" or "deny access" from the auth module.
323 In our scenario, we want to open the door for a few seconds when an access is successful. In case
324 an access is failed, we do nothing.
325 This is easy to do using the doorman module.
326 
327 ~~~~~~~~~~~~~~~~~~~.xml
328 <module>
329  <name>DOORMAN</name>
330  <file>libdoorman.so</file>
331  <level>50</level>
332  <module_config>
333  <instances>
334  <instance>
335  <name>A_DOORMAN_INSTANCE</name>
336  <auth_contexts>
337  <auth_context>
338  <name>AUTH_CONTEXT_1</name>
339  </auth_context>
340  </auth_contexts>
341  <actions>
342  <action>
343  <on>GRANTED</on>
344  <target>my_door_gpio</target>
345  <cmd>
346  <f1>ON</f2>
347  <f2>3000</f3>
348  </cmd>
349  </action>
350  </actions>
351  </instance>
352  </instances>
353  </module_config>
354 </module>
355 ~~~~~~~~~~~~~~~~~~~
356 
357 We are defining an action that will target `my_door_gpio` and send the `ON` command for `3000` ms.
