10 Introduction {#remote_control_intro}
11 ====================================
12 
13 Leosac has a built-in remote control functionality. This means two things:
14  1. The user can interact remotely with Leosac.
15  2. Leosac installation can talk to each other.
16 
17 We aim to provide this functionality in a secure fashion. This means all connections to the
18 remote control socket will be encrypted. We use the CURVE security mechanism from ZeroMQ to
19 reach a good level of security. This security mechanism prevent both eavesdropping and man in the
20 middle attack.
21 Note that this an asymmetric security mechanism and require the use of a private / public keypair.
22 
23 Currently what you can do with remote control is rather limited, and is described in a later
24 section of this document.
25 
26 We will now present the configuration of the remote control feature.
27 
28 **Limitations**: By default there is no authentication of incoming connection yet. Everyone that manages to connect
29 to the remote control socket is allowed to perform ALL available operations. **It is however possible to configure
30 such restrictions.**
31 
32 **Limitations**: When using the Remote Control interface to talk to each other, Leosac units
33 must use the same underlying `boost::serialization` library. Otherwise serialization failure
34 will arise.

36 Configuration Options {#remote_control_user_config}
37 ===================================================
38 
39 If no `<remote>` tag is defined in the configuration, the remote control interface
40 will be disabled.
41 
42 
43 Options | Options | Options | Description | Mandatory
44 --------------|----------|------------------|--------------------------------------------------|-----------
45 port | | | Port to bind the remote control interface to | YES
46 secret_key | | | Z85 encoded secret key | YES
47 public_key | | | Z85 encoded public key | YES
48 security | | | Restrict access to the remote control interface | NO (default to everyone has all access)
49 ---> | map | | Define permission for one user | YES
50 ---> | ---> | pk | Z85 public key of the remote user | YES
51 ---> | ---> | default | Default permission for unspecified command | NO (default to `false`)
52 ---> | ---> | COMMAND_NAME | Name of an existing remote control command. | NO
53 ---> | ---> | COMMAND_NAME_2 | Another existing remote control command. | NO

56 Example {#remote_control_example}
57 ---------------------------------
58 
59 Remote Control configuration takes place in the `<kernel>` tag. It is **not** configured
60 as a module.
61 
62 The following example gives FULL access to anyone that can connect to the remote control socket.
63 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
64 <remote>
65  <port>12345</port>
66  <secret_key>Ygk2EVo#hr*uTG=U[jFFWrb9HDW-V?388=kj)AUz</secret_key>
67  <public_key>TJz$:^DbZvFN@wv/ct&[Su6Nnu6w!fMGHEcIttyT</public_key>
68 </remote>
69 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
70 
71 The following example shows how to enable some restrictions.
72 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
73 <remote>
74  <port>12345</port>
75  <secret_key>Ygk2EVo#hr*uTG=U[jFFWrb9HDW-V?388=kj)AUz</secret_key>
76  <public_key>TJz$:^DbZvFN@wv/ct&[Su6Nnu6w!fMGHEcIttyT</public_key>
77  <security>
78  <map>
79  <!-- Can do everything -->
80  <pk>a_public_key</pk>
81  <default>true</default>
82  </map>
83  <map>
84  <!-- Only allowed the MODULE_CONFIG command -->
85  <pk>an_other_public_key</pk>
86  <default>false</default>
87  <MODULE_CONFIG>true</MODULE_CONFIG>
88  <MODULE_LIST>true</MODULE_LIST>
89  </map>
90  <map>
91  <!-- No default means default to false. Shouldn't be able to do anything -->
92  <pk>again_an_other_pubkey</pk>
93  </map>
94  </security>
95 </remote>
96 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

99 Exposed features {#remote_control_features}
100 ===========================================
101 
102 The set of feature available through the remote control API is rather thin.
103 
104 You can retrieve some information about the running configuration of Leosac:
105 + The `MODULE_LIST` command will return the list of the module currently loaded on the target Leosac unit.
106 + The `GENERAL_CONFIG` command retrieve the global configuration of Leosac.
107 + The `MODULE_CONFIG` command retrieve the configuration for one given module.
108 + The `SYNC_FROM` command order Leosac to fetch and apply the configuration of an other, remote, Leosac unit.
109 + The `SAVE` command order the receiving Leosac to save its current configuration to disk.
110 + The `CONFIG_VERSION` command returns the current serial number of the configuration. This can be
111  used to poll for config update.
112 
113 See below for a detailed description of messages.
114

132 One module name per frame. As many frames as needed. Note that module names are standard and hardcoded in the
133 module code.

135 GENERAL_CONFIG {#remote_control_cmd_general_config}
136 ---------------------------------------------------
137 
138 The `GENERAL_CONFIG` command retrieve Leosac's general configuration. This mean the configuration
139 data except the modules' configuration.
140 
141 It is wise to not export all the configuration. For example, the `<remote>` tag include the server
142 private key. You probably don't want to share this.
143 To control what information are sent when receiving a GENERAL_CONFIG command, see
144 [the sync_source configuration option](@ref sync_source).
145 
146 Frame | Content | Type
147 ---------|------------------------------------------------|-------------------------------------------------------------
148 1 | "GENERAL_CONFIG" | `string`
149 2 | Configuration Type (boost text archive or xml) | `uint8_t`
150 
151 @note: Configuration is an enumeration named [ConfigFormat](@ref Leosac::ConfigManager::ConfigFormat).
152 
153 Response
154 
155 Frame | Content | Type
156 ---------|-----------------------------------------------|-------------------------------------------------------------
157 1 | "OK" | `string`
158 2 | Configuration Content | `string`
159 
160 The response is either a boost text archive, or xml.

195 SYNC_FROM {#remote_control_cmd_sync_from}
196 -----------------------------------------
197 
198 The `SYNC_FROM` command allows Leosac to replicate the configuration of an other Leosac unit.
199 
200 **SECURITY CONSIDERATION**: A module can have additional configuration file. In order to correctly replicate
201 a configuration, we need to copy (and paste) those files too. Leosac will do that. That means that it will
202 **blindly write the content of a received file where it is told to**. That means that syncing from a
203 malicious unit can cause to overwrite **any** file in your filesystem (if Leosac runs as root).
204 
205 From Client to Server:
206 
207 Frame | Content | Type
208 ---------|---------------------------------------------------|---------------------------------
209 1 | "SYNC_FROM" | `string`
210 2 | The endpoint to sync from: "tcp://IP:PORT" | `string`
211 3 | Autocommit (aka writing conf to disk) `1` or `0` | `uint8_t`
212 4 | Remote Leosac server key | `string`
213 5 | Also sync general config (`1` or `0`). | `uint8_t`
214 
215 If the 3rd frame has a value of `1`, we will write the new configuration to disk as soon as the
216 synchronisation is done.
217 
218 The 4th frame is the z85 encoded public key of the remote Leosac server. This is an additional
219 security to make sure you are connecting to the server expect to connect to.
220 
221 The 5th frame is a flag to tell if the general config should also be sync.
222 If not set, only modules list (and their configuration) would be synchronised.
223 If set to `1`, the general configuration will also be synchronized (honoring `sync_from` and `sync_dest`).
224 
225 Syncing general config has no effect unless autocommit is set to `1`. The reason
226 is that sync general config will trigger Loesac's restarts. So if the configuration
227 wasn't autocommit'd it will be lost.
228 
229 Notes: If only the modules (and their config) are synchronised, Leosac will reload them on the fly. However,
230 if the general configuration data (network, logger cfg, remote control configuration) are to be synchronized to,
231 Leosac will restart in order to apply the changes.
232 
233 
234 From Server to Client (if **no error occurred**):
235 
236 Frame | Content | Type
237 ---------|--------------------------|------------
238 1 | "OK" | `string`
239 
240 From Server to Client (in case **something went wrong**):
241 
242 Frame | Content | Type
243 ---------|---------------------------------------|--------------------------
244 1 | "KO" | `string`
245 2 | Reason | `string`

247 SAVE {#remote_control_save}
248 ---------------------------
249 
250 When `SYNC_FROM` configuration, the new configuration is applied directly but it is
251 not saved to disk, unless the `<autosave>BOOLEAN</autosave>` configuration option is set to `true`.
252 
253 The `SAVE` command will make Leosac save its current configuration to disk.
254 
255 From Client to Server:
256 
257 Frame | Content | Type
258 ---------|-----------------------------------------|-------------------
259 1 | "SAVE" | `string`
260 
261 
262 From Server to Client:
263 
264 Frame | Content | Type
265 ---------|---------------------------------|------------
266 1 | "OK" | `string`
267 
268 It is possible the `SAVE` command fails, if that's the case, the response will
269 be 2 frames:
270 
271 Frame | Content | Type
272 ---------|---------------------------------|------------
273 1 | "KO" | `string`
274 2 | "Some reason of why it fails." | `string`
