10 Introduction {#mod_auth_file_intro}
11 ===================================
12 
13 The module Auth-File is an **authentication backend** module.
14 It listens to events generated by **authentication source** module (like `Wiegand`). These events
15 contains credentials. This module verify the credential and either grant or deny access.
16 Granting or denying access means sending a message to the application. This module doesn't open a door
17 or perform action on its own.
18 
19 
20 @note This module is suitable only for a low number of users / cards / groups.
21 The whole configuration lives in memory so this is where this limitation comes from.
22 
23 
24 Configuration Options {#mod_auth_file_user_config}
25 ===================================================
26 
27 Options | Options | Description | Mandatory
28 -----------|-------------|-----------------------------------------------------------------------|-----------
29 instances | | List of configured auth file instance | YES
30 ---> | name | Configure of one particular led device | YES
31 ---> | auth_source | Which device (auth source) we listen to. Can appear multiple times. | YES
32 ---> | config_file | Path to the config file that holds permissions data | YES
33 ---> | target | Name of the target (door) that we are authenticating against | NO
34 
35 Notes:
36  + If the `target` is not present, the module assumes the default target, and will ignore target-specific
37 permissions.
38  + the `config_file` path is relative to the working directory of Leosac.
39  + You can enter multiple `auth_source` device. The module instance will listen to all of them.
40 
41 @warning The `target` field is prefixed by the instance name and a dot when checking for permission
42 in the permission configuration file. This makes it easier to synchronize: you put all
43 configuration in the permission file and simply sync everything, the name of the door will
44 adapt based on the instance name. Assuming `instance_name` being `rpi-1` and `target` being
45 `door1` the matching name in the permission file shall be `rpi-1.door1`.

48 Configuration reload {#mod_auth_cfg_reload}
49 ============================================
50 
51 If you made modification to the **module's configuration file** it is possible
52 to request reloading of this config without having to restart Leosac.
53 Sending `SIGHUP` to the Leosac process will trigger an asynchronous
54 reload of the configuration: during the time it takes to load the new
55 configuration, the old configuration is still used.

57 Users {#mod_auth_user}
58 ======================
59 
60 Users are obviously a core component of an authentication module. The users
61 will map to group, credentials, etc.
62 Users can hold additional property, like their name, email address, or
63 the status of their account (`enabled`, or `disabled`).
64 
65 Declaring the users {#mod_auth_user_declare}
66 --------------------------------------------
67 
68 This section is about describing how to declare users.
69 This takes place in the additional config file.
70 
71 Options | Options | Options | Description | Mandatory
72 --------|---------|----------------|---------------------------------------------------------|-----------
73 users | | | List of users we declare | YES
74 ---> | user | | Declaration of one user | YES
75 ---> | ---> | name | Globally unique name to identify a user. | YES
76 ---> | ---> | firstname | The firstname of the user. | NO
77 ---> | ---> | lastname | The family name of the user | NO
78 ---> | ---> | email | An email address for the user. | NO
79 ---> | ---> | validity_start | Date when the user starts becoming valid / enabled | NO
80 ---> | ---> | validity_end | Date when the user becomes invalid / disabled | NO
81 ---> | ---> | enabled | A boolean to fully enable or disable an account | NO (defaults to `true`)
82 
83 The datetime format for `validity_{start,end}` is this: `%d/%m/%Y %H:%M` (see man (3) `strptime`).
84 
85 ### Example {#mod_auth_user_declare_example}
86 
87 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
88 <users>
89  <user>
90  <name>edgar_hoover</name>
91  <firstname>Edgar</firstname>
92  <lastname>Hoover</lastime>
93  <email>eh@fbi.gov</email>
94 
95  <validity_start>25/10/2015 17:05</validity_start>
96  </user>
97 </users>

100 Credentials Type {#mod_auth_cred}
101 =================================
102 
103 A credentials maps to a user.
104 
105 3 types of credentials are supported:
106  1. WiegandCard
107  2. PINCode
108  3. WiegandCardPin
109 
110 All credentials type share 4 optional properties that you can configure per credentials:
111  1. `id`: The id of the credentials. Used when mapping credentials to schedules. This can be whatever you want,
112  as long as it is unique.
113  2. `enabled`: Is the card active? If true, the credentials can be used, otherwise it will look like it
114  doesn't exist.
115  3. `validity_start`: Optional date at which the credentials becomes active.
116  4. `validity_end`: Optional date at which the credentials stops being active
117 
118 
119 It is possible to omit the `id` field. It is necessary only if you plan to map credentials directly to
120 schedules, instead of mapping user to schedules. The direct mapping of credentials to schedules can be used
121 to create temporary cards with more privilege than the user usually have.
122 
123 WiegandCard {#mod_auth_cred_wiegandcard}
124 -----------------------------------------
125 
126 This credentials is simply a Wiegand card number. It is qualified through a string
127 representing the card id, and an integer representing the number of meaningful bits in
128 the card id.
129 
130 Options | Options | Description | Mandatory
131 -----------------|----------------|-----------------------------------------------------------------------|------------
132 WiegandCard | | Defines a WiegandCard credential | NO
133 ---> | card_id | The id of the card, text string (`aa:bb:cc:dd`) | **YES**
134 ---> | bits | Number of meaningful bits | **YES**
135 
136 
137 PINCode {#mod_auth_cred_pincode}
138 ---------------------------------
139 
140 This credentials is simply a PIN code. The number of character in the code is variable.
141 
142 Options | Options | Description | Mandatory
143 -----------------|----------------|-----------------------------------------------------------------------|------------
144 PINCode | | Defines a PINCode that will be a credentials | NO
145 ---> | pin | The PIN code to be used. This is a string. | **YES**
146 
147 
148 WiegandCardPin {#mod_auth_cred_wiegandcard_pin}
149 ------------------------------------------------
150 
151 This is the combination of a PIN code a WiegandCard. This is currently the most complex credential type.
152 
153 Options | Options | Description | Mandatory
154 -----------------|----------------|-----------------------------------------------------------------------|------------
155 WiegandCardPin | | Defines a credentials composed of a card and a PIN code | NO
156 ---> | pin | The PIN code to be used. This is a string. | **YES**
157 ---> | card_id | The id of the card, text string (`aa:bb:cc:dd`) | **YES**
158 ---> | bits | Number of meaningful bits | **YES**
159 
160 
161 Example {#mod_auth_cred_example}
162 --------------------------------
163 
164 Below an example for each of those credentials:
165 
166 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
167 <credentials>
168  <map>
169  <user>llamasticot</user>
170  <WiegandCard>
171  <card_id>aa:bb:cc:dd</card_id>
172  <bits>32</bits>
173  <enabled>false</enabled>
174  </WiegandCard>
175  </map>
176  <map>
177  <user>MY_USER</user>
178  <PINCode>
179  <id>my_unique_id_for_this_cred</id>
180  <pin>1234</pin>
181  </PINCode>
182  </map>
183  <map>
184  <user>Toto</user>
185  <WiegandCardPin>
186  <card>cc:dd:ee:ff</card>
187  <pin>1234</pin>
188  </WiegandCardPin>
189  </map>
190 </credentials>

193 Groups {#mod_auth_group}
194 ===================================
195 
196 The module also supports groups. A user can be in multiple groups, and will collect
197 permissions from all the groups he's a member of.
198 
199 There is no definition of groups, they are lazily created when mapping users to them.
200 
201 Mapping user to group {#mod_auth_user_to_group}
202 -----------------------------------------------
203 
204 A `<group_mapping>` block can be used to map user to groups.
205 
206 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
207 <group_mapping>
208  <map>
209  <group>Admins</group>
210  <user>MY_USER</user>
211  <user>Toto</user>
212  </map>
213 </group_mapping>
214 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
215 
216 **Note**: It is *not* possible to map credentials directly to group.

219 Schedules {#mod_auth_schedule}
220 =============================
221 
222 Schedules can be defined. They represents time-frame where access **is granted**.
223 Schedules are then mapped to user or to group.
224 
225 They can apply to any `auth_target` (aka doors) or they can target specific doors
226 only. This is configured when mapping a schedule, so that the same schedule
227 can be better reused.
228 
229 See below for an example of how to declare schedules.
230 
231 Declaring schedules {#mod_auth_sched_declare}
232 ---------------------------------------------
233 
234 Options | Options | Options | Options | Description | Mandatory
235 ------------------|-------------|------------|---------|-------------------------------------------------------|-----------
236 schedules | | | | We declare our schedules here | YES
237 ---> | schedule | | | Description of on schedule entry | YES
238 ---> | ---> | name | | A unique name for a schedule | YES
239 ---> | ---> | "weekday" | | A day of the week (`monday`, `sunday`, ...) | NO
240 ---> | ---> | ---> | start | Start hour (format is `hh:mm`) | NO
241 ---> | ---> | ---> | end | End hour (format is `hh:mm`) | NO
242 
243 
244 ### Example {#mod_auth_sched_declare_example}
245 
246 This is a simple example that declares one schedule, named `my_schedule`. It would grants access
247 anytime on monday, and wednesday from 11h to 13h and from 17h to 19h.
248 
249 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
250 <schedules>
251  <schedule>
252  <name>my_schedule</name>
253  <monday>
254  <start>00:00</start>
255  <end>23:59</end>
256  </monday>
257  <wednesday>
258  <start>11:00</start>
259  <end>13:00</end>
260  </wednesday>
261  <wednesday>
262  <start>17:00</start>
263  <end>19:00</end>
264  </wednesday>
265  </schedule>
266 </schedules>
267 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

269 Schedules Mapping {#mod_auth_schedule_map}
270 ------------------------------------------
271 
272 Schedules can, or rather must, be referenced by name. They can be mapped to either
273 `users`, `groups`, or directly to `credentials`.
274 
275 Options | Options | Options | Description | Mandatory
276 ------------------|-------------|------------|-----------------------------------------------------------------------|-----------
277 schedules_mapping | | | Defines the mapping between schedule and user/group/we | YES
278 ---> | map | | One mapping entry | YES
279 ---> | ---> | schedule | Which schedule are we mapping | YES
280 ---> | ---> | user | To which user are we applying the schedule(s) | NO
281 ---> | ---> | group | To which group are we applying the schedules(s) | NO
282 ---> | ---> | credential | To which credential (identified by its `id`) do we apply the schedule | NO
283 ---> | ---> | door | Name of the auth target the mapping apply to | NO
284 
285 Note that `schedule`, `user` and `group` can be used multiple times in the same mapping entry.
286 
287 
288 ### Example {#mod_auth_schedule_map_example}
289 
290 There is 2 mappings entry in the following example.
291  1. The first maps 2 schedules, `name_of_schedule` and `name_of_another_schedule` and applies them
292  to the user `My_User`, `A_Second_User` and to anyone who is a member of the group `the_whole_group_has_the_schedule`.
293  2. The seconds enable the `sysadmin_access_schedule` to the `sysadmins` group when the authentication
294  attempt target the `sysadmin_room`. See the `target` field in the [module options](@ref mod_auth_file_user_config).
295 
296 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
297 <schedules_mapping>
298  <map>
299  <schedule>name_of_schedule</schedule>
300  <schedule>name_of_another_schedule</schedule>
301  <user>My_User</user>
302  <user>A_Second_User</user>
303  <group>the_whole_group_has_the_schedule</group>
304  </map>
305  <map>
306  <schedule>sysadmin_access_schedule</schedule>
307  <group>sysadmins</group>
308  <door>sysadmin_room</door>
309  </map>
310 </schedules_mapping>
311 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
