7 Introduction {#general_config_intro}
8 ====================================
9 
10 Even though Leosac is heavily based on modules, there are general
11 configuration options. Those are defined in the main configuration file,
12 outside of the `<modules>` and `</modules>` tags.
13 
14 Here is an overview of what we call "general configuration":
15  + Database configuration
16  + Logger configuration
17  + Network configuration
18  + Remote control configuration.
19 
20 
21 Path Management {#general_config_path_mng}
22 ==========================================
23 
24 Leosac has to know 2 important paths:
25  1. The path to the factory configuration directory: this can be set with the environment variable
26  `LEOSAC_FACTORY_CONFIG_DIR`.
27  2. Paths to where Leosac's shell scripts are saved. Customizable with `LEOSAC_SCRIPTS_DIR` env variable.

30 Instance Name {#general_instance_name}
31 ==========================================
32 
33 Each unit is given an `instance_name` to help identify it when running in a environment
34 with multiple Leosac system. This field is mandatory.
35  1. This `instance_name` information is never synchronized.
36  2. It is used to automatically prefix some configuration option: for example
37  the `target` (door) in the auth file module. This allows easier config synchronization.

40 Database {#general_database}
41 ============================
42 
43 Leosac can work with either an SQLite or PostgreSQL database. The database is mandatory to provide
44 support for some features. For example, the WebSocket API needs access to the
45 database to store users, permissions, etc. Similarly, the HardwareService needs the
46 DBService to operate.
47 
48 The `database` tag holds various information about the database.
49 
50 Options | Options | Description | Mandatory
51 --------------|----------|--------------------------------------------------------|-----------
52 type | | Underlying database type. Either "sqlite" or "mysql". | YES
53 path | | **SQLite only**: Path to the database file. | YES if SQLite
54 username | | **PGSQL only**: Database username. | YES if MySQL
55 password | | **PGSQL only**: Password for the database user. | YES if MySQL
56 dbname | | **PGSQL only**: Database name to use. | YES if MySQL
57 host | | **PGSQL only**: Database hostname / IP. | NO
58 port | | **PGSQL only**: Port the database listens to | NO
59 
60 Example {#database_example}
61 --------------------------
62 Below is an example for SQLite.
63 
64 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
65 <database>
66  <type>SQLite</type>
67  <path>leosac.sqlite</path>
68 </database>
69 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
70 
71 An one for PGSQL.
72 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
73 <database>
74  <type>pgsql</type>
75  <username>root</username>
76  <password></password>
77  <dbname>leosac</dbname>
78 </database>
79 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

81 Automatic Configuration Saving {#general_config_save}
82 =====================================================
83 
84 There is a useful configuration option: `autosave`. When set to true,
85 the current configuration of Leosac will be saved to disk when Leosac exits.
86 It defaults to false.
87 
88 Logger Configuration {#general_config_logger}
89 =============================================
90 
91 Leosac has a builtin logger for information and / or debugging purposes. It can write to syslog.
92 It is not recommended to log DEBUG message to syslog.
93 
94 
95 Configuration Options {#logger_user_config}
96 -------------------------------------------
97 
98 Default configuration will log everything to `stdout`. Entry with
99 a level >= `WARNING` will be written to syslog.
100 
101 Options | Description | Mandatory
102 ---------------|----------------------------------------------------|-----------
103 enable_syslog | Enable logging to syslog | NO (default to `true`)
104 enable_database| Enable logging to the configured (if any) database.| NO (default to `false`)
105 min_syslog | Minimal log entry level to write to syslog | NO (default to `WARNING`)
106 
107 Here is a list of the various log level available:
108  + `DEBUG`
109  + `INFO`
110  + `NOTICE`
111  + `WARNING`
112  + `ERROR`
113  + `CRITICAL`
114  + `ALERT`
115  + `EMERGENCY`
116 
117 Example {#logger_example}
118 -------------------------
119 
120 Logger configuration takes place in the `<kernel>` tag. It is **not** configured
121 as a module.
122 
123 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
124 <log>
125  <enable_syslog>true</enable_syslog>
126  <min_syslog>INFO</min_syslog>
127 </log>
128 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

131 Network Configuration {#general_config_network}
132 ================================================
133 
134 Leosac can provide an abstraction to network configuration.
135 You can configure your IP address, netmask and gateway in Leosac's configuration file.
136 
137 This allows the Remote Control API (or Rpleth API) to change the network configuration
138 of the Leosac unit.
139 
140 However, if you do not plan on changing your IP address or see no use for this feature,
141 it is recommended to use Linux tools to configure your network instead (ie, write to `/etc/network/interfaces`).
142 
143 @note Disabling the network subsystem (with `<enabled>false</enabled>`) doesn't disable the network. It disables
144 the network management that Leosac does. If you want to fully disable the network, you'll have to do this
145 at the operating system level.
146 
147 @note CIDR notation is not supported (see [#54](https://github.com/leosac/leosac/issues/54)).
148 
149 Example {#network_example}
150 --------------------------
151 Network configuration takes place in the `<kernel>` tag. It is **not** configured
152 as a module.
153 
154 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
155 <network>
156  <enabled>false</enabled>
157 </network>
158 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
159 
160 An other example with enabled network management.
161 
162 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
163 <network>
164  <enabled>true</enabled>
165  <interface>eth0</interface>
166  <dhcp>true</dhcp>
167  <netmask>255.255.255.0</netmask>
168  <default_ip>192.168.0.18</default_ip>
169  <gateway>192.168.0.1</gateway>
170 </network>
171 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

179 Sync Source {#sync_source}
180 ==========================
181 
182 From all these configuration options, you may want to limit which one gets
183 exported over the network when a remote command to fetch the global configuration is received.
184 
185 If no `<sync_source>` is present, export everything.
186 Here is a list of exportable config tag:
187  + `<autosave>`
188  + `<plugin_directories>`
189  + `<log>`
190  + `<network>`
191  + `<remote>`
192  + `<sync_dest>`
193 
194 Note that the `no_import` tag cannot be exported.
195 
196 Example {#sync_source_ex}
197 -------------------------
198 
199 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
200 <sync_source>
201  <log>true</log>
202  <network>false</network>
203  <remote>false</remote>
204 </sync_source>
205 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
206 
207 Sync Dest {#sync_dest}
208 ======================
209 
210 This is similar to `sync_source` but is on the receiver side.
211 The tag set to true in `sync_dest` are the tag that will be updated with the content from the remote server.
212 The rules are the same than `sync_source`.
213 
214 Non Importable Module {#non_importable_module}
215 ===============================================
216 
217 By default all modules are synchronised when syncing configuration - be it when the Replication module
218 runs, or when the server receive the `SYNC_FROM` remote control command.
219 It's however possible to prevent module configuration from being added, erased or updated.
220 
221 To do so, define a `<no_import>` tag in the config file.
222 
223 **Note**: The `<no_import>` tag will **NEVER** be synchronized from a remote server.
224 
225 *Security notice:* This is because it is not exported, not because we carefully ignore it when importing.
226 
227 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~.xml
228 <no_import>
229  <TEST_AND_RESET />
230  <ANOTHER_MODULE />
231 </no_import>
232 ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
233 
234 This will have the effect of leaving alone the configuration of Test and Reset and Another Module.
235 It will not prevent the module config from being exported.

237 Version {#version}
238 ==================
239 
240 A `version` field is configurable. This is mostly used when running a master/slave
241 configuration.
242 If you make a change to the configuration, and increments this `version` field,
243 slave should automatically update their configuration.
